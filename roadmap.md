# Marrakech (Mars Theme) boardgame.io 実装ロードマップ

この文書は、既存の HTML 実装を捨てて `boardgame.io` ベースでマルチプレイ対応のゲームとして再構築するための実装ロードマップである。
主用途は **Codex に参照させながら段階的に実装を進めること**。

---

## 0. ゴール

### 目的

* 既存の Marrakech 風ボードゲームを、`boardgame.io` を用いた構成で再実装する。
* 友人同士が各デバイスからアクセスし、同一ルームで対戦できるようにする。
* まずは **ローカル起動 / 小規模プレイ / 最小機能完成** を優先し、その後に UI 改善やデプロイを行う。

### 最終到達イメージ

* ブラウザからアクセスできる
* 3 人対戦が可能
* ルームを作成して参加できる
* 1 ターンの流れがゲーム仕様どおりに動作する
* 盤面、所持金、アッサム位置、向き、フェーズ、ログが同期される

---

## 1. 採用技術方針

### 基本構成

* ゲームエンジン: `boardgame.io`
* フロントエンド: `React`
* 言語: `TypeScript` を推奨
* 開発環境: `Vite`
* スタイリング: 最初は最低限の CSS
* 配置先: 後で検討（最初はローカル）

### 方針

* **ゲームロジックと UI を明確に分離**する
* 状態の正は `boardgame.io` 側に持たせる
* UI は `G` / `ctx` を描画し、moves を呼ぶだけに寄せる
* まずは **1 画面で完成する最小構成**を目指し、後から演出や見た目を改善する

## 2. 最初に固定する仕様

実装着手前に、曖昧さをなくすために以下を明文化する。

### 2.1 プレイヤー

* プレイヤー数は 3 人固定で開始する
* プレイヤーIDは `0 / 1 / 2` とし、表示上は `A / B / C`

### 2.2 盤面

* 六角形マップ（7 段）
* 行ごとのマス数: `4-5-6-7-6-5-4`
* 座標系を固定する

  * 例: `(row, col)`
  * row は `0..6`
  * col は各行の有効範囲のみ存在
* 隣接判定の仕様を座標系込みで固定する

### 2.3 地形

* 地形タイプ: `sea | mountain | city`
* 各マスは以下のいずれか

  * 空
  * 地形あり（type, owner を持つ）
* 上書き配置を許可する

### 2.4 アッサム

* 初期位置: `(3,3)`
* 初期向き: `北東`
* 向きは 6 方向とする
* 盤外に出る場合の「盤端を沿う折り返し」の仕様を厳密化する

### 2.5 ターンの流れ

1. 向き変更
2. 移動
3. 支払い判定
4. 配置（2 マス）
5. 手番終了

### 2.6 移動

* 毎ターン 1〜3 マスをランダム移動
* ランダム値はゲームログに記録する
* リプレイ性やデバッグのため、乱数生成は game ロジック内で一元管理する

### 2.7 支払い

* 着地先が他プレイヤー所有の地形なら支払い発生
* 支払い額 = 着地地形と **同タイプかつ同オーナーで連結している枚数**
* 支払先 = 地形オーナー
* 資金不足時の扱いを先に決める

  * 借金を許すか
  * 0 までしか払えないか
  * 脱落があるか
* 初版では「所持金が負になってもよい」または「持っている分だけ払う」のどちらかに固定する

### 2.8 配置

* 1 手番で同じ地形タイプを 2 マス置く
* 1 マス目:

  * アッサム隣接マスから選択
* 2 マス目:

  * 1 マス目に隣接
  * アッサム位置ではない
* 既存地形の上書きを許可
* 1 手番中に配置する 2 マスの地形タイプは同一

## Phase 1. プロジェクト雛形作成

### 目的

boardgame.io + React の最小起動環境を作る。

### タスク

* Vite + React + TypeScript プロジェクト作成
* `boardgame.io` の導入
* クライアント起動確認
* ローカルで 2〜3 タブ接続できることを確認

### 完了条件

* 開発サーバーが起動する
* 仮のゲーム状態が画面に表示される
* moves を1つ呼べる

### Codex 指示例

* React + TypeScript + Vite の雛形を作成
* boardgame.io を導入
* 最小サンプルゲームを作成し、ローカルで複数クライアント接続できる状態にする

---

## Phase 2. ドメインモデルの定義

### 目的

ゲームロジック実装前に、データ構造を固める。

### タスク

* 型定義を作る

  * `PlayerId`
  * `TerrainType`
  * `Direction`
  * `HexCoord`
  * `Tile`
  * `AssamState`
  * `GamePhase`
  * `GameLogEntry`
  * `MarrakechState`
* 初期状態生成関数を作る
* 初期配置を反映する

### 完了条件

* `setup()` で仕様どおりの初期状態が作られる
* State を console / UI 上で確認できる

### Codex 指示例

* Marrakech 用の型定義を TypeScript で作成
* 初期盤面、初期所持金、初期アッサム状態、初期地形配置を setup に実装

---

## Phase 3. 盤面座標と隣接判定の実装

### 目的

六角形盤面の基本操作を先に完成させる。

### タスク

* 有効マス判定 `isValidCell(coord)`
* 全マス列挙 `getAllCells()`
* 隣接マス列挙 `getNeighbors(coord)`
* 向きに応じた次マス計算
* テストケース作成

### 重要ポイント

* このフェーズの品質が低いと以後すべて壊れる
* 先にユーティリティとして切り出す
* 盤面ロジックは UI から独立させる

### 完了条件

* 任意マスの隣接マスが正しく取れる
* 六角形盤面で盤外判定が正しい

### Codex 指示例

* 7 段六角形マップ専用の座標ユーティリティを実装
* 有効マス判定と隣接判定に対してテストを書く

---

## Phase 4. フェーズ管理と手番進行

### 目的

1 ターンの流れを boardgame.io の moves / phases / turn で表現する。

### タスク

* フェーズ定義

  * `chooseDirection`
  * `moveAssam`
  * `placeFirstTile`
  * `placeSecondTile`
* 各フェーズで使える move を制限
* 手番終了時に次プレイヤーへ回す
* HUD 用の派生情報を計算する

### 完了条件

* 正しい順番でしか操作できない
* A → B → C の順に手番が回る

### Codex 指示例

* boardgame.io の phase 機能を使って 1 ターンの進行を実装
* 各 phase で許可される move を制限し、不正操作を弾く

---

## Phase 5. 向き変更の実装

### 目的

アッサム隣接マス選択による向き変更を実装する。

### タスク

* 隣接マスクリックから方向を決定する move を作る
* 選択可能マスのハイライト情報を返す
* UI 側で選択候補を表示する

### 完了条件

* 隣接マスを選ぶと向きが変わる
* その後 move フェーズに遷移する

### Codex 指示例

* アッサム隣接マスを選択して direction を更新する move を実装
* 隣接以外のマス指定は拒否する

---

## Phase 6. 移動と折り返し処理の実装

### 目的

Marrakech の核となるアッサム移動を完成させる。

### タスク

* 1〜3 のランダム歩数決定
* 1 歩ずつ進めるロジック実装
* 盤外時の折り返し処理実装
* 最終位置と最終向きの更新
* ログ出力

### 注意

* ここは仕様の解釈差分が出やすい
* 先に examples を複数作る
* 「この位置・向きから何歩でどこへ行くか」のテストを書く

### 完了条件

* 盤中央・辺・角付近で期待どおり移動する
* ログに歩数と最終位置が記録される

### Codex 指示例

* 六角盤上のアッサム移動ロジックを実装
* 盤外時は盤端を沿うように折り返す仕様を関数化
* 代表ケースのユニットテストを追加

---

## Phase 7. 支払い判定の実装

### 目的

着地時の支払い処理を実装する。

### タスク

* 着地マス取得
* 着地マスが他人所有か判定
* 同タイプ・同オーナー連結成分サイズを BFS / DFS で計算
* 所持金の増減処理
* ログ出力

### 完了条件

* 支払いが必要なときのみ発生する
* 連結サイズに応じて金額が正しく変わる
* 所持金が更新される

### Codex 指示例

* 着地マスから同タイプ同オーナーの連結成分サイズを求める関数を実装
* 支払い処理を move フローに組み込み、ログに記録する

---

## Phase 8. 配置フェーズの実装

### 目的

1 手番で同タイプ 2 マス配置する流れを実装する。

### タスク

* 現在選択中の地形タイプを state に保持
* 1 マス目配置 move
* 2 マス目配置 move
* 妥当性チェック

  * 1 マス目はアッサム隣接
  * 2 マス目は 1 マス目隣接
  * 2 マス目はアッサム位置不可
* 上書き配置処理
* 手番終了処理

### 完了条件

* 仕様どおり 2 マス配置できる
* 不正配置は拒否される
* 手番終了後に次プレイヤーへ移る

### Codex 指示例

* 配置フェーズを 2 段階 move に分けて実装
* 1 マス目と 2 マス目で制約条件を分離して検証する
